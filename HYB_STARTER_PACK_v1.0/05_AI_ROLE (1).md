# Project Rules

## 1. 작업 단위
- 한 번에 하나의 명확한 목표만 수행한다.
- 변경 전 현재 상태를 확인한다.
- 변경 후 실행 또는 테스트로 성공 여부를 검증한다.
- 성공을 확인한 뒤 다음 단계로 이동한다.

## 2. 코드 제공 방식
- 사용자가 교체할 파일은 가능한 한 전체 파일 코드로 제공한다.
- 파일 경로를 코드 위에 명시한다.
- 여러 파일을 수정할 경우 수정 순서를 번호로 제공한다.
- 생략 표시(`...`)가 있는 코드를 완성본처럼 제공하지 않는다.

## 3. 설명 방식
각 작업은 다음 순서로 안내한다.

```text
왜 하는지
→ 어느 파일을 수정하는지
→ 전체 코드
→ 저장 방법
→ 실행 명령어
→ 정상 결과 예시
→ 오류 발생 시 확인할 항목
```

## 4. 아키텍처 원칙
- 외부 API 코드는 비즈니스 계산 로직과 분리한다.
- 마켓별 응답을 내부 공통 모델로 변환한다.
- 설정값과 비밀키를 코드에 직접 넣지 않는다.
- 계산 규칙은 한 곳에서 관리한다.
- UI는 핵심 엔진을 직접 복제하지 않고 서비스 계층을 호출한다.
- DB 접근 로직은 화면과 분리한다.
- 테스트 가능한 작은 함수와 모듈을 선호한다.

## 5. 품질 기준
- 실행된다고 완료로 판단하지 않는다.
- 예외 처리, 입력 검증, 로그, 테스트를 함께 고려한다.
- 숫자 계산에는 근거와 단위를 명확히 한다.
- 외부 API 응답이 비어 있거나 실패할 수 있음을 전제로 한다.

## 6. Git 규칙
권장 브랜치 흐름:

```text
main
└─ feature/<작업명>
```

권장 커밋 예시:

```text
feat: add opportunity fee calculation
fix: handle empty ebay search result
refactor: separate marketplace adapter
docs: update current project status
test: add opportunity engine cases
```

커밋 전 확인:

- 비밀키가 포함되지 않았는지
- `.env`가 추적되지 않는지
- 실행되지 않는 임시 파일이 없는지
- 변경된 문서가 있는지
- 테스트 또는 직접 실행이 성공했는지

## 7. 의사결정 기록
중요한 구조 변경은 이유와 대안을 문서에 남긴다.
